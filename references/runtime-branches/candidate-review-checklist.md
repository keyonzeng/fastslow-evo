# FastSlow Candidate Review Checklist

Use this checklist when reviewing, updating, merging, splitting, or rejecting a promotion candidate.

## Identity
- Is the candidate trying to represent one underlying pattern only?
- Is the canonical problem stated plainly?
- Is the claimed pattern understandable without jargon inflation?

## Evidence Quality
- Are the linked evidence records concrete?
- Do they preserve enough context?
- Is any linked evidence weak, misleading, or only superficially similar?
- Should any evidence be excluded from this candidate?

## Pattern Sameness
- Do these records really share the same root cause?
- Are the differences only wording/source differences?
- Do task goal, user intent, and failure location line up?
- Should this candidate be split instead of expanded?

## Convergence Quality
- Is this candidate better than keeping records separate?
- Is there an existing candidate this should merge into instead?
- Is this candidate duplicating another candidate?
- Should this one be superseded or rejected?

## Promotion Logic
- Is local handling no longer enough?
- Is recurrence real rather than implied by file count?
- Is there a smallest durable artifact that makes sense?
- Is the proposed artifact lighter than a full framework rewrite?

## Validation and Rollback
- Is there a clear validation path?
- Is regression detectible?
- Is rollback practical?
- Is rejection or split criteria clear if later evidence weakens the merge?

## Final Decision
Choose explicitly:
- merge
- keep separate
- split
- supersede
- create
- observe
- reject

Then explain:
- what happened
- why these records do or do not belong together
- why this next action is justified now
