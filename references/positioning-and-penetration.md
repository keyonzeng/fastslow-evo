# Positioning and Penetration

## Core Positioning

FastSlow Evo is a **dual-loop self-evolution system**:
- it decides what should be fixed quickly
- it decides what should be promoted slowly
- it converts local experience into durable capability through ongoing cycles

This is the definition of what it **is**.

Do not collapse this into a narrower platform-only description.

## Penetration Strategy

Penetration is a separate question:
- who should adopt first
- where feedback quality is highest
- where repeated usage is most likely
- where fast/slow routing creates immediate value

This is the definition of who it should **win first**.

## Principle

- **Positioning should stay broad.**
- **Penetration should stay focused.**

## First-Wave Users

### 1. OpenClaw power users

Why:
- persistent workspace and memory
- repeated interactions across sessions
- messaging plus tooling plus delivery in one system
- strong need for “be more like me over time”

These users benefit most from:
- correction-driven fast adaptation
- behavior tuning across chat surfaces
- durable memory and workflow promotion

### 2. Builders and workflow designers

Why:
- failure signals are frequent and visible
- repeated workflows produce clear recurrence patterns
- tiny fixes create immediate value
- promotion into scripts/skills/workflows is natural

These users benefit most from:
- fast hardening after tool and workflow failures
- slow promotion into reusable capabilities

## Second-Wave Users

### 3. Heavy users of other agent environments

Examples:
- Claude Code
- Cursor
- OpenCode

Why:
- they can reuse the fast/slow method
- they can adopt templates, checklists, and promotion logic
- but they do not define the primary runtime assumptions

## Third-Wave Users

### 4. Mainstream office and study users

Why not first:
- low tolerance for ceremony
- weaker structured feedback loops
- they mainly want visible output improvement, not meta-governance

How they benefit:
- consume packs, templates, review loops, and curated starter workflows
- use outputs of the system before using its full mechanism

## Product Rule

Design the core for the first-wave users.
Design lightweight entry points for the later-wave users.

Do not weaken the core to chase broad adoption too early.
