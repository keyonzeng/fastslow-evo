# FastSlow Evo 使用说明（给 OpenClaw 用户）

## 这是什么

FastSlow Evo 是一个让 OpenClaw 智能体**边做事、边变得更稳**的工作方法。

它不靠脚本自动乱改自己，也不把每次小问题都升级成长期规则。
它做的是一件更重要的事：

**判断什么只需要本地小修，什么值得慢慢沉淀成长期能力。**

你可以把它理解成一个双环机制：

- **Fast loop（快环）**：局部小修，马上改善下一次
- **Slow loop（慢环）**：重复出现、已经稳定的问题，进入长期晋升审查

---

## 它适合解决什么问题

FastSlow Evo 特别适合这些场景：

- 同类问题反复出现，但你不想每次都重新提醒
- 某些输出经常“看起来不错，但不够可执行”
- 你希望助手慢慢贴近你的风格、标准和工作方式
- 你希望改进是可解释、可回看、可纠错的，而不是黑箱自我进化

典型例子：

- 会议总结总漏 action items
- 回复语气太官方
- 工具结果偶尔读错
- 某类工作流总在同一个地方出错
- 有些修正只该局部生效，有些则该沉淀成长期规则

---

## 它不是什么

FastSlow Evo 不是：

- 自动改 prompt 的黑箱系统
- 一套 Python 自动化脚本
- 看到问题就乱建规则的“自我改造机”
- 把每一次 annoyance 都升级成 skill 的借口

它的核心不是自动化，而是：

**由 OpenClaw 主模型基于上下文做判断。**

---

## 先理解 5 个状态

FastSlow Evo 默认会把一个信号判断成以下 5 种状态之一：

### 1. fast
适合：
- 局部问题
- 下次很快能验证
- 小改一下就能明显改善

例如：
- 这次回复太官方，下次自然一点
- 这次摘要漏了一个 action item

### 2. slow candidate
适合：
- 这个问题不是第一次了
- 已经有重复证据
- 不再只是本地小修能解决
- 值得进入长期规则/能力的评审

例如：
- 多次会议总结都漏 action items
- 同类工具总结总出现相同遗漏

### 3. observe
适合：
- 你怀疑这里有问题
- 但证据还不够
- 现在就做长期规则，容易过拟合

例如：
- 一次模糊投诉
- 一次说不清是不是风格偏好

### 4. ignore
适合：
- 噪音
- 一次性小情绪
- 没有长期复用价值

### 5. reject
适合：
- 某个已有 candidate 其实不成立
- 之前 merge 错了
- 这个方向不该继续保留

这个状态很重要，因为系统不只需要“新增”，也要能“关掉错误沉淀”。

## OpenClaw 用户怎么用

最简单的用法不是运行命令，而是**在聊天里明确表达你的意图**。

### 你可以直接这样说

#### 让它做快环
- 记住这个修正，下次别再这样
- 这次走 fast loop，给我最小修复
- 以后这种情况这样处理

#### 让它判断该快还是慢
- 你判断这个该 fast 还是 slow
- 这个问题老是重复，你来判断怎么处理
- 你看这是不是值得沉淀成长期规则

#### 让它做慢环候选审查
- 这个模式已经重复多次，进入 slow candidate review
- 看看这几个 evidence 是不是同一个 pattern
- 帮我判断这些记录该 merge 还是 split

#### 让它做 heartbeat review
- 用 FastSlow Evo review 一下最近 evidence
- 看看最近有没有值得升级的 candidate
- 检查一下有没有该 reject 或 split 的 candidate

## 实际工作流怎么走

### 场景 1：出现一个新的修正或问题

做法：

1. 写一条 evidence 记录
2. 保留完整上下文
3. 让 OpenClaw 判断它属于 fast / slow candidate / observe / ignore / reject
4. 只做最小必要动作

重点：

- 先保留 evidence
- 不要过早下结论
- 不要看到一次问题就升级长期规则

---

### 场景 2：你怀疑两个记录其实是同一个问题

做法：

1. 读取两个 evidence 的完整上下文
2. 比较：
   - 任务目标是否一致
   - 用户意图是否一致
   - 失败位置是否一致
   - 根因是否一致
3. 再判断：
   - same-pattern
   - related-but-distinct
   - split-needed
   - unclear

重点：

**不能只看关键词像不像。**

“都提到了 missing action items” 不代表就是同一个 pattern。
一个可能是会议总结问题，另一个可能是项目规划问题。
症状像，不等于根因一样。

---

### 场景 3：你想让它做长期沉淀

前提不是“这看起来挺重要”，而是：

- 有重复证据
- 是同一个 underlying pattern
- 本地小修已经不够
- 能说清怎么验证
- 能说清如果错了怎么回滚

只有这样，才适合进入 `slow candidate`。

重点：

**Slow 不是奖励，Slow 是审查门槛更高。**

---

## evidence 怎么写才有用

一条 evidence 最少应该包含：

- **Summary**：发生了什么
- **Source**：来自用户、运行过程、回顾还是外部内容
- **Context**：这是在什么场景下发生的
- **Full Context**：任务目标、用户意图、前后对话、已有行为
- **Raw Signal**：原始信号或原话
- **Notes**：轻量备注，比如 gap、recurrence_hint

重点是：

**evidence 是证据，不是结论。**

不要在 evidence 阶段就写成：
- 这一定是某某 pattern
- 这肯定要 promote
- 这一定和前面一样

这些都应该留给后面的 review 判断。

---

## candidate 是干什么的

candidate 不是日志，也不是批准结果。

它的作用是：

**把多个 evidence 收敛成一个“值得审查的长期问题对象”。**

一个好的 candidate 应该能说清：

- 到底在重复发生什么
- 为什么这些 evidence 是一回事
- 为什么 local fix 不够了
- 最轻的长期方案是什么
- 怎么验证是否有效
- 如果判断错了，怎么 split / reject / rollback

如果这些说不清，这个 candidate 就不够强。

---

## heartbeat 应该怎么理解

heartbeat 不是自动晋升器。

它的作用是：

- 回看最近 evidence
- 看看有没有 pattern 正在稳定
- 看看有没有 candidate 该更新
- 看看有没有 candidate 该 split 或 reject
- 如果没有实质变化，就 no-action

重点：

**heartbeat 的价值在 review，不在热闹。**

如果 heartbeat 每次都产生一堆新文件，通常说明系统在制造噪音，而不是在成长。

---

## 判断时最重要的 4 个问题

如果你懒得记全部协议，就记这 4 个：

### 1. 到底发生了什么？
不要先抽象，先落地。

### 2. 为什么这件事值得记？
是质量问题、执行问题、风格问题，还是边界问题？

### 3. 这和之前真的是同一个 underlying pattern 吗？
不能只看表面像不像。

### 4. 下一步最小必要动作是什么？
是：
- fast
- slow candidate
- observe
- ignore
- reject

不是所有问题都值得沉淀成长期能力。

---

## 什么时候适合 fast，什么时候适合 slow candidate

### 更适合 fast
- 局部问题
- 风格修正
- 单次遗漏
- 下次很快能验证
- 改动小且低风险

### 更适合 slow candidate
- 已经重复出现
- 同一个 pattern 比较稳定
- 涉及重复工作流或更大边界
- 本地规则已经不够
- 可以写出验证路径

### 更适合 observe
- 证据太薄
- 还分不清是不是同一个问题
- 贸然沉淀容易过拟合

### 更适合 reject
- candidate 其实站不住
- merge 逻辑不成立
- 这个方向不该继续占用系统注意力

## 给 OpenClaw 用户的最短建议

如果你现在就想开始用，只做这三件事：

### 第一步
遇到值得记的问题，就写 evidence，不要只留在聊天里。

### 第二步
让 OpenClaw 按协议判断：
- fast
- slow candidate
- observe
- ignore
- reject

### 第三步
默认优先做最小动作，不要过度晋升。

也就是：

**先变准，再变重。**

---

## 一句话总结

FastSlow Evo 的本质，不是让智能体“自动进化”，而是让它在 OpenClaw 里：

**基于真实 evidence，有纪律地判断什么该马上修，什么该慢慢沉淀，什么根本不该留下。**
