# OpenClaw Fit

FastSlow Evo is not defined by OpenClaw, but OpenClaw is the strongest first host for it.

## Why OpenClaw fits unusually well

OpenClaw combines:
- persistent workspace files
- memory layers
- direct messaging surfaces
- multi-session continuity
- tool usage
- skill loading
- delivery back into the active chat surface

This creates the exact environment where dual-loop evolution is useful.

## OpenClaw-native fast-loop targets

Prioritize fast adaptation for:
- user corrections in chat
- repeated formatting or tone mistakes
- tool-call summarization errors
- delivery mismatches
- repeated omissions in summaries, actions, or follow-ups
- high-frequency workflow friction

Why:
These are local, frequent, and visible. They should improve within the same day or next few similar tasks.

## OpenClaw-native slow-loop targets

Prioritize slow promotion for:
- memory write policy
- shared vs private context boundaries
- behavior rules across group/private/direct surfaces
- durable workflow rules
- capability and validation specs
- promotion into reusable skills

Why:
These changes have larger blast radius and must be validated before becoming durable.

## Important Distinction

OpenClaw is the primary host context, not the total product definition.

The method should remain portable.
The defaults should remain OpenClaw-native.
